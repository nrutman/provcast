# Contributing to Provcast

This document defines the code style, conventions, and quality standards for the Provcast codebase. All contributors (human and AI) should follow these guidelines.

## Naming Conventions

### Casing Rules

| Entity | Casing | Examples |
|--------|--------|---------|
| Functions | `camelCase` | `formatDuration`, `loadAudio`, `handleClick` |
| React components | `PascalCase` | `AudioControls`, `StepSidebar`, `FileSelectStep` |
| Classes and types | `PascalCase` | `AudioInfo`, `Metadata`, `WizardStep` |
| Constants | `UPPER_SNAKE_CASE` or `camelCase` | `CBR_BITRATES`, `defaultMetadata` |
| Files | **Match the primary export** | `AudioControls.tsx`, `useAudioStore.ts`, `cn.ts` |

**File naming matches the export:**
- `AudioControls.tsx` → `export function AudioControls()`
- `useAudioStore.ts` → `export const useAudioStore = ...`
- `cn.ts` → `export function cn()`

**Exceptions:**
- **shadcn/ui components** (`app/components/ui/`) use kebab-case (`button.tsx`, `card.tsx`) since they're generated by the shadcn CLI.
- **Tauri domain modules** (`app/hooks/tauri/`) use lowercase (`playback.ts`, `editing.ts`) since they group multiple exports by domain rather than having a single named export.
- **Rust files** follow Rust convention: `snake_case` for files and functions, `PascalCase` for types and structs.

### TypeScript

| Entity | Casing | Example |
|--------|--------|---------|
| Files | Match export | `AudioControls.tsx` |
| Functions | `camelCase` | `formatDuration()` |
| Components | `PascalCase` | `function AudioControls()` |
| Interfaces/types | `PascalCase` | `interface AudioInfo` |
| Variables | `camelCase` | `const sampleRate = 44100` |
| Zustand stores | `camelCase` (use-prefix) | `useAudioStore` |
| React hooks | `camelCase` (use-prefix) | `useKeyboardShortcuts` |

### Rust

| Entity | Casing | Example |
|--------|--------|---------|
| Files | `snake_case` | `processor.rs`, `mod.rs` |
| Functions | `snake_case` | `fn find_quietest_region()` |
| Structs/enums | `PascalCase` | `struct AudioEngine` |
| Constants | `UPPER_SNAKE_CASE` | `const FRAME_SIZE: usize` |
| Variables | `snake_case` | `let sample_rate = 44100` |

## File Organization

### One Primary Export Per File

Each file should have a single primary export, and the file should be named after that export:

```
formatDuration.ts   →  export function formatDuration() { ... }
AudioControls.tsx   →  export function AudioControls() { ... }
useAudioStore.ts    →  export const useAudioStore = ...
```

**Exceptions:**
- Type definitions closely related to the primary export may be co-exported (e.g., a store's state interface alongside the store itself).
- Thin Tauri command wrappers may be grouped by domain (e.g., `playback.ts` exports `playAudio`, `pauseAudio`, `stopAudio`).

**Avoid** catch-all files like `utils.ts` or `helpers.ts`. If a utility is used in multiple places, give it its own file named after what it does.

### Directory Structure (Namespaces)

Each directory acts as a namespace. Organize its contents by visibility:

| Subdirectory | Purpose |
|-------------|---------|
| *(top level)* | Public members — the primary exports of this namespace |
| `utils/` | Private helpers shared across files in the namespace |
| `hooks/` | React hooks used within the namespace |
| `errors/` | Custom Error classes used within the namespace |
| `__tests__/` | Tests for the public members |

```
app/components/steps/           # Namespace: wizard step components
  FileSelectStep.tsx            # Public: step component
  NormalizationStep.tsx         # Public: step component
  TrimmingStep.tsx              # Public: step component
  MetadataStep.tsx              # Public: step component
  ExportStep.tsx                # Public: step component
  utils/                        # Private: shared helpers
    formatTime.ts               # Shared time formatting functions
  __tests__/                    # Tests
    FileSelectStep.test.tsx
    NormalizationStep.test.tsx
```

**Guidelines:**
- Helpers used by only one file should stay **inline** in that file (at the bottom, per source code ordering)
- Helpers shared across multiple files in the same namespace go in `utils/`
- React hooks that serve a namespace go in `hooks/` (distinct from plain utility functions)
- Tests always go in `__tests__/`, mirroring the source file name

**Rust equivalent:** Rust uses `mod.rs` for public API, private functions within modules, and `#[cfg(test)] mod tests` inline. No separate `utils/` directory is needed since Rust's module privacy handles this natively.

### Source Code Ordering

Within each file, code should be ordered as follows:

1. **Imports**
2. **Types and constants** needed by the primary export (interfaces, type aliases, config constants)
3. **The primary export** (the main function, component, or class)
4. **Private helpers** (sub-components, utility functions, internal types)

For **React component files**, this means the exported component appears before its private sub-components and helpers. Function declarations are hoisted in JavaScript, so sub-components defined below can be referenced above:

```tsx
// 1. Imports
import { useState } from "react";

// 2. Types/constants for the export
interface StepSidebarProps { ... }
const steps = [...];

// 3. Primary export
export function StepSidebar({ currentStep, onStepClick }: StepSidebarProps) {
  return <div>{steps.map(s => <StepButton key={s.id} step={s} />)}</div>;
}

// 4. Private helpers (hoisted — safe to use above)
function StepButton({ step }: { step: Step }) { ... }
```

For **test files**, the same principle applies:

1. **Imports and mocks** (`vi.mock`, test fixtures)
2. **`describe` block** (the main test suite)
3. **Helper functions** (factories, assertion utilities)

### Test File Location

Test files live in `__tests__/` subdirectories alongside the modules they test:

```
app/stores/useAudioStore.ts
app/stores/__tests__/useAudioStore.test.ts

app/components/steps/ExportStep.tsx
app/components/steps/__tests__/ExportStep.test.tsx
```

Integration tests live in `app/test/integration/`.

### README Files

Add `README.md` files to directories where the architecture isn't obvious from reading the code. Use [Mermaid](https://mermaid.js.org/) for diagrams. Don't repeat implementation details that are clear from the source — focus on:

- **Why** things are organized this way
- **How** components interact (data flow, boundaries)
- **What** constraints or invariants exist

Skip README files for directories where the code is self-explanatory.

## TypeScript Conventions

### Imports

- Use the `@/` path alias for imports from `app/` (e.g., `import { Button } from "@/components/ui/button"`)
- Group imports: external libraries, then `@/` imports, then relative imports

### Components

- Use `function` declarations for React components (not arrow functions)
- shadcn/ui components live in `app/components/ui/` — don't modify them unless necessary
- Icons come from `lucide-react` or `react-material-symbols`

### State Management

- Zustand stores live in `app/stores/`
- Store types (interfaces) are co-exported with the store
- Prefer granular selectors: `useStore((s) => s.field)` over `useStore()`

### Tauri IPC

- Frontend command wrappers live in `app/hooks/tauri/` grouped by domain
- Each wrapper is a thin async function that calls `invoke()` and handles store updates
- Audio data never crosses the IPC boundary — only metadata, peak data, and file paths

## Rust Conventions

### Module Organization

- Each module (`mod.rs`) should have a clear single responsibility
- Public types that cross the IPC boundary go in `commands.rs` with `#[derive(Serialize, Deserialize)]`
- Internal types stay in their domain module

### Code Ordering

Same principle as TypeScript:

1. **Imports and `use` statements**
2. **Public types** (structs, enums used by callers)
3. **Public functions** (the module's API)
4. **Private helpers**
5. **Tests** (`#[cfg(test)] mod tests { ... }`)

### Error Handling

- Tauri commands return `Result<T, String>` — convert errors with `.map_err(|e| e.to_string())`
- Use `parking_lot::Mutex` for state synchronization (not `std::sync::Mutex`)
- Drop locks before performing I/O or emitting events

### Documentation

- Use `///` doc comments on all public functions and types
- Include parameter descriptions for non-obvious parameters
- Document invariants and safety requirements (especially for `unsafe` code)

## Testing

### Test Quality Checklist

Every time tests are added or updated, evaluate:

1. **Are there high-value cases we're missing?** — Error paths, state transitions, edge cases, integration points between modules.
2. **Can the tests be simplified or consolidated?** — Duplicated setup, parameterizable assertions, overly verbose tests.
3. **Are there low-value tests to remove?** — Tests that check implementation details instead of behavior, tautological assertions (e.g., `typeof fn === "function"`), tests that duplicate what other tests already cover.

### What to Test

- **Behavior, not implementation** — Test what a function does, not how it does it.
- **User interactions** in component tests — Click buttons, type in fields, verify results. Don't just check that elements render.
- **Error handling** — What happens when Tauri commands fail? When files are missing?
- **State transitions** — Store updates, cross-store consistency, reset behavior.
- **Integration points** — Components + stores, frontend + Tauri commands (via mocked `invoke`).

### What Not to Test

- That React renders children (this tests React, not your code)
- That exports exist (the import would fail if they didn't)
- Default values of state (test behavior that depends on defaults instead)
- Trivial arithmetic (unless there are boundary conditions)

### Test Patterns

```typescript
// GOOD: Tests behavior
it("clicking delete removes the selected region", async () => {
  // setup: select a region
  // action: click delete
  // assert: region is gone, store updated
});

// BAD: Tests implementation detail
it("renders a button with text 'Delete'", () => {
  render(<Component />);
  expect(screen.getByText("Delete")).toBeInTheDocument();
});
```

## Git Conventions

- Branch names: `feature/description`, `fix/description`, `refactor/description`
- Commit messages: imperative mood, describe the "why" not just the "what"
- One logical change per commit
- Run `bash check.sh` before pushing (TypeScript compile, lint, format, tests, Rust fmt, clippy, tests, build)
